# GitHub 双平台自动部署设计

## 目标

从本次调整开始，项目唯一开发目录为：

```text
D:\work\code\code_o\github_ye\enjoylifeyay
```

GitHub 仓库 `yakoye/enjoylifeyay` 作为唯一源码来源。推送 `main` 后，自动完成质量检查，并分别发布到：

- 正式主站：`https://enjoylifeyay.goldke.online/`
- Cloudflare 平台别名：`https://enjoylifeyay.pages.dev/`
- GitHub Pages 备用站：`https://yakoye.github.io/enjoylifeyay/`

Cloudflare Pages 保留为正式站，GitHub Pages 用于静态备份和部署兜底。两个站点都只发布静态内容，不再提供评论功能。

## 当前状态与约束

- Astro 使用静态输出，构建目录为 `dist/`。
- Cloudflare 项目 `enjoylifeyay` 是 Direct Upload 项目，不能直接转换成 Git 集成项目。
- 现有仓库包含评论前端、Pages Functions、D1 SQL 和管理脚本，本轮将完整移除这些评论功能代码。
- GitHub Pages 项目站部署在 `/enjoylifeyay/` 子路径，现有根路径链接和图片路径必须适配该前缀。
- 覆盖复制保留了旧目录中已经删除的文件，必须在提交前清理，防止重复内容重新进入站点。
- 不向 GitHub 提交 `.env`、Cloudflare Token、账户缓存或其他密钥。

## 总体架构

```text
本地修改
  -> git push origin main
    -> GitHub Actions: verify
      -> Cloudflare 构建（根路径）
        -> Wrangler Direct Upload
          -> enjoylifeyay.pages.dev
      -> GitHub Pages 构建（/enjoylifeyay/）
        -> Pages Artifact
          -> yakoye.github.io/enjoylifeyay/
```

两次部署从同一个提交构建，但使用不同的部署目标参数，避免把 GitHub Pages 的子路径带入 Cloudflare 主站。

## 构建目标与环境变量

| 字段 | Cloudflare | GitHub Pages |
| --- | --- | --- |
| `DEPLOY_TARGET` | `cloudflare` | `github-pages` |
| `BASE_PATH` | `/` | `/enjoylifeyay/` |
| 公开站点地址 | `https://enjoylifeyay.goldke.online` | `https://yakoye.github.io/enjoylifeyay` |
| 评论 | 不提供 | 不提供 |

站内路径通过统一的 URL 工具函数生成。Markdown 中以 `/` 开头的内部链接和图片路径在 GitHub Pages 构建时添加 `BASE_PATH`，外部链接、锚点、邮箱地址和协议链接保持不变。

正式 canonical 统一指向 Cloudflare 自定义域名 `https://enjoylifeyay.goldke.online` 的对应页面，避免 Cloudflare 平台别名和 GitHub Pages 被搜索引擎视为独立内容。

## GitHub Actions

工作流在拉取请求和 `main` 推送时运行：

1. 使用 Node.js 24 和 `npm ci` 安装锁定依赖。
2. 执行 `npm run verify`，包括 Astro 检查、测试、内容审计、构建和内部链接检查。
3. 只有 `main` 推送且验证成功后才执行部署。
4. Cloudflare 任务重新进行根路径生产构建，并通过 Wrangler 上传现有项目。
5. GitHub Pages 任务使用 `/enjoylifeyay/` 构建，上传 Pages Artifact，再由官方 Pages Action 发布。

部署任务不得绕过验证任务。任一构建失败时，对应部署停止，已上线版本保持不变。

## Cloudflare 凭据

GitHub 仓库需要以下 Actions Secrets：

| Secret | 用途 | 公开性 |
| --- | --- | --- |
| `CLOUDFLARE_ACCOUNT_ID` | 指定 Cloudflare 账户 | 仅 Actions 使用 |
| `CLOUDFLARE_API_TOKEN` | 向 `enjoylifeyay` 执行 Pages 部署 | 仅 Actions 使用 |

Token 只授予 Cloudflare Pages 部署所需的最小权限，不写入仓库、README、日志或构建产物。

## 评论功能清理

两个部署目标都删除评论入口和评论服务，具体范围包括：

- 删除文章页面中的评论组件引用。
- 删除评论组件与浏览器端提交脚本。
- 删除 `functions/api/comments.js`，Cloudflare 部署恢复为纯静态站点。
- 删除评论管理脚本及 `package.json` 中的评论命令。
- 删除仓库内仅服务于评论功能的 SQL、迁移文件和现行配置说明。
- 更新 README、部署文档和内容维护文档，不再引导配置 D1 或评论审核。

Cloudflare 账户中已有的评论 D1 数据库也在本轮删除。执行顺序固定为：先发布不含评论代码的新版本并确认线上页面不再请求评论 API，再核对数据库名称与绑定，最后删除对应 D1 资源。

## GitHub Pages 行为

- 导航、文章、图片、CSS、脚本、RSS、站点地图和 Pagefind 搜索均使用子路径。
- 文章页不显示评论区域，也不会请求评论 API。
- GitHub Pages 作为备用站，不配置与 Cloudflare 相同的自定义域名。
- GitHub 仓库 Pages Source 设为 `GitHub Actions`。

- 本轮提交包含从原开发目录迁移过来的已确认站点更新、两篇文章、评论功能清理、部署工作流和文档；不包含本地缓存与临时构建产物。

## README 维护流程

README 需要让以后更新可以按下面的固定流程完成：

```powershell
cd D:\work\code\code_o\github_ye\enjoylifeyay
git pull --ff-only
npm ci
npm run verify
git add <本次修改的文件>
git commit -m "docs: update site content"
git push origin main
```

README 同时说明：

- 两个线上地址及主次关系。
- 常用内容目录和新文章 Frontmatter。
- 自动部署触发条件与 GitHub Actions 查看位置。
- Cloudflare Secrets 仅首次配置。
- 自动部署失败时先看 Actions 日志，不重复提交或覆盖线上项目。
- 本地 Wrangler 发布仅作为应急回退，不作为日常流程。

## 验证与验收

提交前必须取得以下新鲜证据：

1. `npm run verify` 完整通过。
2. Cloudflare 根路径构建通过。
3. GitHub Pages 子路径构建通过。
4. 构建产物检查确认导航、文章、图片和搜索没有错误根路径。
5. 两篇新增文章通过内容 schema、目录和引用检查。
6. 构建产物和源码不再包含评论表单、`/api/comments` 请求或评论管理命令。
7. Cloudflare 上对应的评论 D1 数据库已在无评论版本上线后删除。
8. Git 状态不包含密钥、`.wrangler/` 缓存或构建产物。
9. 推送 `main` 后 GitHub Actions 成功。
10. Cloudflare 主站和 GitHub Pages 备用站核心页面均返回 HTTP 200。

## 失败处理

- 验证失败：不部署，修复后重新执行完整验证。
- Cloudflare Secret 缺失：GitHub Pages 可以发布，Cloudflare 任务明确失败并提示首次配置，不回退到提交密钥。
- GitHub Pages 路径错误：停止发布该产物，修复 `BASE_PATH` 后重建。
- 科学文章引用无法核实：文章保持草稿，不删掉问题标记后强行公开。
- D1 名称或用途无法确认：停止删除并继续核对，不删除任何其他数据库。
- 自动部署异常：保留当前线上版本；必要时使用已验证的 `dist/` 手工执行 Wrangler 应急发布。

## 非目标

- 不迁移或替换现有 Cloudflare 项目。
- 不引入 React、SSR、数据库、CMS 或额外前端框架。
- 不让两个托管平台同时绑定同一个自定义域名。
